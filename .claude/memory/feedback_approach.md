name: 작업 스타일 및 피드백
description: 사용자의 작업 선호도 및 코드 리뷰 피드백
type: feedback
---

- 본 프로젝트는 팀 프로젝트입니다.

- 작업 순서는 다음을 따르세요.
코드 수정 → 빌드 확인 → 사용자 로컬 테스트 OK → 커밋

- 문서나 문자열 같은 간단한 수정은 main에 직접 푸시 가능합니다.

- 프로젝트에서 기능별 작업 시 GitHub 이슈나 마일스톤을 먼저 생성하세요.

- 이후 이슈별 브랜치/PR 워크플로우를 따르세요.
이슈 먼저 생성 → 브랜치 생성 → 구현 → 로컬 테스트 후 커밋 → PR → 사용자가 직접 머지

- 다음 규칙을 따르세요.
  - 브랜치명: `issue/번호-간략설명`
  - 같은 파일 수정하는 연관 이슈는 하나의 브랜치/PR로 묶어도 됨
  - PR 머지 후 다음 이슈 브랜치 생성 (스택 방지)
  - 우선순위 순서대로 진행 (이슈 번호 순서 아님)

- Issues 혹은 Pull Requests 생성 시 labels를 붙이세요. (적절한 labels가 없으면 생성할 것)

- 더 이상 필요가 없어진(useless) 폴더, 파일, 코드는 주기적으로 정리하세요.

- git add 시 CLAUDE.md는 제외하세요.
